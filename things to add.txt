1. Internal Delta Time Handling

- Instead of making the user calculate and pass dt every frame, the window wrapper should own the clock.

- The Clock Instance:
    - Move pygame.Clock() inside the Framex class.

- Automatic Scaling:
    - Create a method that calls .tick() and stores the result as a property (e.g., self.dt).

- Encapsulation:
    - Objects can simply reference the Framex instance to see the current dt, keeping the main loop clean.


2. Rebuild Sprite Grouping

- Shift from manual list-checking to Pygame's built-in Sprite and Group system for better performance and organization.

- Specialized Groups:
    - Use pygame.sprite.Group() for standard objects and pygame.sprite.LayeredUpdates() if you want to handle "Z-order" (drawing things behind or in front of others).

- Batch Drawing:
    - The Framex update method should accept a group and call group.draw(surface). This replaces complex isinstance checks with optimized internal C code.


3. Debug Tools & Callbacks

- A "test window" needs to show you what is happening behind the scenes.

- Visual Hitboxes:
    - Add a toggle that draws rect outlines. This helps verify if a sprite's image matches its physical boundaries.

- Performance Overlay:
    - Use a system font to render clock.get_fps() in the corner of the screen.

- Collision Callbacks:
    - Instead of just stopping a sprite, add a "hook." When a collision is detected, the sprite can trigger a function (e.g., on_hit()) to handle specific logic like taking damage or bouncing.


4. Keymap Injection

- Hardcoding "WASD" or "Arrows" limits the user. By allowing a keymap, the user can define controls in the main script.

- Dictionary Mapping:
    - The Entity should accept a dictionary where keys are actions and values are Pygame constants (e.g., {"up": pygame.K_w}).

- Logic Decoupling:
    - The input() method checks the dictionary values rather than specific keys, making it easy to swap controls mid-game.


5. Collision Correction

- Your current logic "centers" the sprite on the object it hits, which causes it to get stuck or teleport.

- Edge Snapping:
    - When a collision occurs on the X-axis, the player's right side should snap to the obstacle's left (if moving right), or vice versa.

Two-Phase Movement:
- Move on X → Check X collisions → Move on Y → Check Y collisions.
    - This prevents the "diagonal clipping" bug.


6. Static vs. Dynamic Distinction

- Separating objects into these two categories optimizes your physics engine.

- Static Entities:
    - These represent the environment (walls, floors). They are added to a collision_group but never move. They don't need update() or input() methods, saving CPU cycles.

- Dynamic Entities:
    - These are the "Actors" (player, NPCs). They move and check for collisions against the Static group. This prevents "everything-against-everything" collision checks, which can lag a game as it grows.
